1. Project Overview
The Secondhand Vehicle Web Service is a full-stack, multi-panel web application designed to digitize and streamline the buying, selling, and renting of used vehicles and drivers. The platform serves three user groups — administrators who manage all operations, buyers who browse and purchase vehicles, and customers who rent vehicles or hire drivers — and now adds a built-in referral program that rewards customers for bringing in new buyers.
This revision updates the original specification in two ways: (1) the backend has moved from Django (Python) to Node.js with Express, and the database is standardized on PostgreSQL only (MySQL is no longer supported); (2) a new Referral & Commission system has been added so that buyers can generate a unique, shareable referral link for any vehicle, and if the referred person purchases that vehicle, the system automatically tracks the conversion, notifies the admin via the Status Event Log, and surfaces the referrer's contact details for commission payout.

2.3.2 Referral Relationship Rules
CUSTOMER → REFERRAL (referrer_customer_id): one customer can generate many referral links, one per vehicle they choose to share
VEHICLE → REFERRAL (vehicle_id): a referral link always targets exactly one vehicle, so a customer generates a distinct link per vehicle they want to refer
REFERRAL.referrer_customer_id and REFERRAL.referred_customer_id both reference CUSTOMER — a self-referential pair on the same table, enforced so referrer_customer_id ≠ referred_customer_id
REFERRAL → SALE_ORDER (sale_order_id): set only when the referred customer's purchase of the exact referred vehicle is confirmed by the admin
REFERRAL → ADMIN_USER (processed_by_admin_id): set when an admin approves or marks the commission as paid
REQUEST_LOG.referral_code (nullable FK to REFERRAL.referral_code): carries the referral attribution from the buy request through to the Admin Accept action
STATUS_EVENT_LOG entries of type referral_converted reference entity_type = 'REFERRAL' and entity_id = referral_id, surfaced on the Admin Status Log alongside referrer name, phone, and commission amount

VEHICLE → VEHICLE_PRICE: one vehicle has multiple pricing records (listing_mode, effective_from/to)

4.2.7 Referral Conversion Logic (Backend Workflow)
Step 1 — Link generation: a logged-in customer on a vehicle detail page taps “Refer & Earn”. Frontend calls POST /api/referrals/generate with the vehicle_id. Backend generates a unique referral_code via nanoid, inserts a Referral row (status = generated), and returns the full shareable link.
Step 2 — Sharing: the customer shares the link via WhatsApp, SMS, or copy-link using the Web Share API / react-share.
Step 3 — Visit tracking: when anyone opens the link, the frontend reads the ?ref= query param, stores it in referralStore (Zustand, persisted to sessionStorage), and calls GET /api/referrals/resolve/:code, which sets status = visited and visited_at = NOW() on first visit.
Step 4 — Attribution at signup or request: if the visitor signs up, the stored referral_code is sent with POST /api/auth/signup and saved to Referral.referred_customer_id (only if not already set, and referred_customer_id ≠ referrer_customer_id). If the visitor already had an account, the referral_code instead travels with POST /api/requests when they submit a buy request for that vehicle, and is stored on REQUEST_LOG.referral_code.
Step 5 — Conversion detection: when an admin accepts the buy request (POST /api/requests/:id/accept) or directly creates a sale order (POST /api/orders/sale), the backend looks up REQUEST_LOG.referral_code (or the Referral row by referred_customer_id + vehicle_id). If a matching Referral exists with status in (generated, visited), the service updates it to status = converted, sets converted_at, links sale_order_id, and calculates commission_amount (price will be entry manually by admin).
Step 6 — Admin notification: the same transaction writes a STATUS_EVENT_LOG entry with event_type = referral_converted, entity_type = 'REFERRAL', entity_id = referral_id, and a payload containing the referrer's full name, phone, email, the referred customer's name, the vehicle, the sale price, and the calculated commission. admin_notified_at is stamped. This appears immediately on the Admin Status Log feed (4.3.2, Page 1) with a dedicated badge.
Step 7 — Commission payout: an admin opens the Referral & Commission page (4.3.2, Page 5), reviews the referrer's contact details, and calls PATCH /api/admin/referrals/:id/commission to mark the commission approved, then paid — each transition is recorded in ACTIVITY_LOG.
Expiry: a node-cron job (referralLinkExpiry.js) runs daily and marks any Referral row past its expires_at (default 90 days) as expired if it has not converted, freeing the customer to generate a fresh link for the same vehicle.

7. Key Notes & Conventions
All timestamps stored in UTC in the database; converted to local time in the frontend
All prices stored as DECIMAL(10,2) in PostgreSQL; displayed in frontend with INR (₹) formatting
PostgreSQL is the only supported database in this revision — all MySQL-specific configuration, drivers, and fallback code paths have been removed
Admin JWTs are separate from Customer JWTs — different issuer claim in payload
Every admin write action (POST/PATCH/DELETE) must write to ACTIVITY_LOG with old_data/new_data JSON
Vehicle listing_mode field must be: "sale", "rental", or "both" — drives which panel shows it
Exchange orders are recorded as two SALE_ORDERs linked by a nullable exchange_ref_id field
OTP is hashed before storage (use bcrypt.hash / bcrypt.compare)
A node-cron task runs every 15 minutes to check overdue rentals and write STATUS_EVENT_LOG
DAILY_ADMIN_SUMMARY is generated by a node-cron task at midnight using DB aggregation
Referral codes are unique, URL-safe, and generated server-side with nanoid — never client-generated
A customer cannot refer themselves: referrer_customer_id and referred_customer_id are validated to differ before a conversion is accepted
One vehicle can have only one active (non-expired, non-converted) referral link per referrer; requesting a new link for the same pair returns the existing active link instead of creating a duplicate
Referral conversion is decided strictly in the backend at the moment an admin confirms a sale — never on the frontend — to prevent spoofed conversion claims
Commission percentage is configurable via the REFERRAL_COMMISSION_PERCENT environment variable so business rules can change without a code deploy
All referral-to-sale linkage is auditable: REQUEST_LOG.referral_code, REFERRAL.sale_order_id, and STATUS_EVENT_LOG together provide a full trail from link share to commission payout
